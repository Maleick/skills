# Codebase Structure

**Analysis Date:** 2026-02-25

## Directory Layout

```
[project-root]/
├── skills/                    # Skill catalog organized by maturity
│   ├── .system/               # Built-in core skills (creator/installer)
│   ├── .curated/              # Maintained public skills (30 directories)
│   └── .experimental/         # Incubating skills and prototypes
├── README.md                  # Project overview and install guidance
├── contributing.md            # Community/security contribution notes
├── .gitignore                 # Local ignore rules
└── .planning/                 # Generated planning artifacts (created during GSD runs)
```

## Directory Purposes

**`skills/.system/`:**
- Purpose: Core skills bundled by default.
- Contains: `skill-creator` and `skill-installer` packages.
- Key files: `skills/.system/skill-creator/SKILL.md`, `skills/.system/skill-installer/SKILL.md`, and their `scripts/` directories.

**`skills/.curated/`:**
- Purpose: User-installable curated skills.
- Contains: 30 skill directories including `skills/.curated/openai-docs/`, `skills/.curated/playwright/`, and `skills/.curated/security-best-practices/`.
- Key files: one `SKILL.md` per skill plus optional `agents/openai.yaml`, `references/`, `scripts/`, `assets/`.

**`skills/.experimental/`:**
- Purpose: Experimental/in-progress skill work.
- Contains: `agent-team-protocol`, `auto-memory`, `cobaltstrike-rest-api`, `ralph-wiggum-loop`.
- Key files: mixed maturity; `skills/.experimental/cobaltstrike-rest-api/SKILL.md` exists while three peers currently lack `SKILL.md`.

**`.planning/`:**
- Purpose: GSD-generated planning state and analysis docs.
- Contains: codebase map artifacts such as `.planning/codebase/*.md`.
- Key files: `.planning/codebase/STACK.md` and peers generated by this workflow.

## Key File Locations

**Entry Points:**
- `README.md`: Human-facing project entry and install examples.
- `skills/**/SKILL.md`: Machine-readable and human-readable skill instruction entrypoints.

**Configuration:**
- `.gitignore`: Repository ignore rules.
- `skills/**/agents/openai.yaml`: UI metadata for skill picker and prompts.

**Core Logic:**
- `skills/.system/skill-installer/scripts/list-skills.py`: Curated/experimental listing logic.
- `skills/.system/skill-installer/scripts/install-skill-from-github.py`: install workflow.
- `skills/.system/skill-creator/scripts/init_skill.py`: new skill scaffolding.
- `skills/.curated/*/scripts/*.py|*.sh|*.js|*.swift`: task-specific automation.

**Testing:**
- Not detected as a dedicated tree (`tests/` absent; no `*.test.*` or `*.spec.*` files found).

## Naming Conventions

**Files:**
- Skill definitions use uppercase `SKILL.md` (for example `skills/.curated/pdf/SKILL.md`).
- Python scripts generally use `snake_case.py` (`skills/.curated/sentry/scripts/sentry_api.py`).
- Shell wrappers use lowercase/snake naming (`skills/.curated/playwright/scripts/playwright_cli.sh`).

**Directories:**
- Skill directories use kebab-case names (`skills/.curated/openai-docs/`, `skills/.curated/security-ownership-map/`).
- Tier directories are dot-prefixed (`skills/.curated`, `skills/.experimental`, `skills/.system`).

## Where to Add New Code

**New Skill:**
- Primary definition: `skills/.curated/<skill-name>/SKILL.md` (or `.experimental/` when incubating).
- UI metadata: `skills/.curated/<skill-name>/agents/openai.yaml`.
- Script helpers: `skills/.curated/<skill-name>/scripts/`.
- Long references: `skills/.curated/<skill-name>/references/`.
- Output assets: `skills/.curated/<skill-name>/assets/`.

**Updates to Core Skill Tooling:**
- Skill scaffolding/validation logic belongs in `skills/.system/skill-creator/scripts/`.
- Install/list behavior belongs in `skills/.system/skill-installer/scripts/`.

## Special Directories

**`skills/.curated/security-best-practices/references/`:**
- Purpose: Large curated security reference corpus.
- Generated: No (maintained docs).
- Committed: Yes.

**`.planning/codebase/`:**
- Purpose: Generated codebase map artifacts.
- Generated: Yes (via GSD mapping workflows).
- Committed: Yes when running commit-enabled map workflows.
